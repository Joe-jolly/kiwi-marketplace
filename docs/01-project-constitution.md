# Kiwi Marketplace

## Project Constitution

Version: 1.0

Status: Approved

---

# 1. Project Identity

Project Name:

Kiwi

Tagline:

Everything starts nearby.

Category:

Marketplace Platform

Primary Market:

Migrants living in South Korea

---

# 2. Mission

Kiwi helps migrants discover nearby items, services, jobs, housing, and opportunities through a simple, trustworthy, and location-based marketplace experience.

The goal is to reduce friction between people who need something and people who can provide it.

---

# 3. Vision

Short-Term Vision:

Build the most useful marketplace for migrants living in South Korea.

Mid-Term Vision:

Become the primary local discovery platform for migrant communities.

Long-Term Vision:

Help people discover opportunities around them, including:

* Products
* Services
* Jobs
* Housing
* Community Opportunities

Everything starts nearby.

---

# 4. Target Audience

Phase 1:

Migrants living in South Korea.

Primary Focus:

* Uzbek speakers
* Russian speakers
* Korean speakers
* English speakers

Phase 2:

Central Asian migrant communities.

Phase 3:

International migrant communities.

---

# 5. Product Philosophy

Kiwi is a marketplace.

Kiwi is not a social network.

Kiwi is not an entertainment platform.

Kiwi is not a content platform.

Every product decision must support:

Post → Chat → Deal

If a feature does not help users complete a transaction faster or easier, it should not be part of the MVP.

---

# 6. Core User Journey

User enters application.

↓

User browses nearby listings.

↓

User searches or filters listings.

↓

User opens a listing.

↓

User starts a chat.

↓

Seller selects a buyer.

↓

Phone number becomes visible.

↓

Deal happens outside the application.

---

# 7. Marketplace Principles

Marketplace First.

Content First.

Location First.

Trust First.

Simple Over Complex.

Working Over Perfect.

Maintainable Over Clever.

---

# 8. Core Marketplace Rules

Rule 1

One post can have many chats.

Rule 2

One participant can have only one chat per post.

Rule 3

Chat is created when the first message is sent.

Rule 4

Seller can select only one chat.

Rule 5

Selected chat changes post status to Reserved.

Rule 6

Selected participant gains access to seller phone number.

Rule 7

Completed posts cannot receive new messages.

---

# 9. Supported Languages

The application must support:

* Uzbek
* Russian
* Korean
* English

Internationalization must be implemented from the beginning.

Hardcoded interface text is prohibited.

All user-facing text must use translation keys.

---

# 10. Supported Currency

Primary currency:

KRW (South Korean Won)

MVP supports only KRW.

Currency conversion is considered a future feature.

---

# 11. Location Principles

Location is a core part of the product.

First launch:

* Request GPS permission
* Detect current location
* Set default marketplace area

Users may manually override location.

Location always remains under user control.

---

# 12. Trust & Safety

Marketplace trust is a critical product requirement.

The platform includes:

* SMS Verification
* User Reports
* Post Reports
* User Blocking
* Human Moderation

AI moderation is not part of MVP.

---

# 13. Guest User Policy

Guests may:

* Browse listings
* Search listings
* View categories
* View post details

Guests may not:

* Create posts
* Start chats
* Save favorites
* Report content
* Access profile features
* Access notifications

Authentication is required for interaction.

---

# 14. Success Metrics

The primary KPI is completed deals.

Secondary metrics:

* Posts Created
* Chats Started
* Reserved Posts
* Returning Users

Metrics such as followers, likes, stories, or entertainment engagement are not product priorities.

---

# 15. Non-Goals

The following are explicitly not MVP goals:

* Social Network Features
* Stories
* Reels
* Livestreaming
* Video Calls
* Voice Calls
* AI Moderation
* AI Search
* Cryptocurrency Features
* Marketplace Gamification

---

# 16. Founder Rule

Before approving any new feature, ask:

"Does this help users move from Post → Chat → Deal?"

If the answer is no, the feature belongs in Future Planning, not MVP.

---

# 17. Constitution Priority

When conflicts occur, the following order applies:

1. Project Constitution
2. Technical Constitution
3. Development Constitution
4. Cursor Operating Manual
5. Feature Requests

The Constitution always wins.
